## вңҰ Key Design Decisions

### Why direct Nvidia NIM `fetch` instead of LangChain wrappers?

The `nv-embedqa-e5-v5` model is **asymmetric** вҖ” it requires `input_type: "passage"` for documents and `input_type: "query"` for questions. LangChain's `NvidiaEmbeddings` abstraction doesn't expose this parameter cleanly, so we use raw `fetch` calls to the NIM API for full control.

### Why `deleteAll()` before each upload?

Single-document context. When a user uploads a new PDF, the old vectors are purged to ensure retrieval stays grounded in the **current** document only. No cross-contamination.

### Why SSE streaming?

Perceived latency. The LLM response streams token-by-token to the client, so the user sees text appearing immediately rather than waiting for the full response.

## вңҰ UI Features

- **Particle wave animation** вҖ” subtle canvas-based dot grid with mouse interaction
- **Drag-and-drop upload** вҖ” with real-time pipeline progress indicators
- **Markdown rendering** вҖ” headers, lists, code blocks, bold text in responses
- **Suggested questions** вҖ” one-click starter prompts after upload
- **Copy to clipboard** вҖ” quick-copy on any AI response
- **Session reset** вҖ” upload a new document without refreshing

## вңҰ API Reference

### `POST /api/upload`

Accepts `multipart/form-data` with a PDF file.

```
Request:  FormData { file: <pdf> }
Response: { success: true, chunks: number, filename: string }
```

Pipeline: `PDF вҶ’ text вҶ’ split (1000ch / 200 overlap) вҶ’ embed вҶ’ upsert to Pinecone`

### `POST /api/chat`

Accepts JSON with a user message.

```
Request:  { message: string }
Response: SSE text/event-stream
```

Pipeline: `Query вҶ’ embed (input_type: "query") вҶ’ top-5 cosine retrieval вҶ’ LLaMA 3.3 70B вҶ’ stream`
